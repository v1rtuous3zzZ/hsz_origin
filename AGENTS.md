# AI 开发约定

本项目当前仅由 AI 协助单一维护者开发；不存在多人并行提交、代码所有权分配或兼容旧分支的需求。实现应优先采用小模块、明确数据模型、可直接运行的命令和可重复验证的测试，避免为多人协作预留无用流程。

## 语言与注释

- 数据库表、字段、索引和存储过程新增或修改时必须提供准确的中文 `COMMENT`，说明业务含义和约束。
- 源码标识符、模块名、CLI 参数、配置键和日志结构使用清晰的英文；代码注释只保留帮助 AI 理解数据来源、幂等性、事务边界和业务限制的内容。
- 文档面向项目维护和 AI 开发，明确输入、输出、验证命令与未确认前提。

## 运行与配置

- Git 仓库必须可通过 README、`.env.example` 和锁文件重建运行环境；真实密码、Token、私钥和生产凭据仍只允许留在本地 `.env` 或部署密钥系统，禁止提交。
- 不提交 `.env`、日志、虚拟环境、`node_modules`、构建产物或本地 Git 历史备份。
- 修改后运行相关测试、Ruff 和实际 CLI dry-run；Git 提交前展示变更摘要。只有维护者明确允许时才推送。

## 运维资料

- `F:\项目运维` 仅用于保存项目运维所需的稳定资料，例如服务器地址、域名、端口、登录方式、账号凭据存放位置、部署路径、服务名、配置文件路径、日志路径、数据库和中间件连接信息、常用启动/停止/重启命令。
- 不要把部署过程、排障过程、脚本启动记录、补数进度、批次结果或临时操作流水写入运维 Markdown；这些内容以实际日志、数据库批次表、systemd 状态或当前对话结论为准。
- 只有当账号、服务器、端口、路径、服务名、配置位置、部署方式等长期运维资料发生变化时，才更新对应运维资料。

## AI 项目说明

- `docs/AI_PROJECT_GUIDE.md` 是后续 AI 了解项目的入口文档。
- 修改接口、认证、数据库表或字段、ETL 同步规则、数据可达范围、部署或运行命令时，必须在同一变更中更新该文档。
- 提交前核对文档的接口清单、数据库职责与验证命令仍与实现一致。

## 前端设计规范

- 所有前端 UI 新建或改动前，必须先阅读 `hsz-origin-system-web/DESIGN.md`，并以其中 IBM Carbon 风格令牌为视觉规范。
- 重点遵循 IBM Blue 主色、IBM Plex Sans、0–4px 方角、无阴影的细边框卡片与扁平企业风格；不得在未更新该文件的情况下另建冲突的视觉体系。

## 门架采集边界

- `t_legacy_gantry_info` 是从旧库复制到项目库的 32 条门架母表，作为实际采集门架的参考基线。
- 远程门架服务器当前只能访问沿江公司 `10.13.*` 网段；不得假设可连接 `10.33.*`、`10.35.*` 等外部网段服务器。
- 当前可获得的出本路段实际交易数据可能只覆盖 G50。外部方向及非 G50 路段必须标记为数据不可达或待确认，不能伪造、补零或把不可连接服务器当作采集失败。
- 中心库的 39 条物理门架可保留用于规则和历史参考；正式采集计划必须按可达服务器与母表基线过滤，并在日志与状态中报告覆盖范围。
- 除生产中心库 `10.13.0.223/hsz_origin` 外，所有 `10.*` 门架源数据库只能执行只读操作（`SELECT`、`SHOW`、`DESCRIBE`、`EXPLAIN`）；严禁在源库执行任何 DDL 或 DML。
- 生产部署时，ODS、命中、事实、批次和断点只允许写入 `10.13.0.223/hsz_origin`。当前本地联调允许写入 `192.168.1.109/hsz_origin`，不得将该地址作为永久生产规则。
